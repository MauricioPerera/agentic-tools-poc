# Threat model

This document is the honest answer to "what stops a malicious skill from
compromising the host running an agent that uses this registry?"

It exists because the second peer-review pass (see commit history) called
out that **`The loader is the trust boundary` was a one-liner where it
needed a real threat model**. Phase 2 (community-contributed skills via
sandboxing) cannot ship without one. Phase 1 (this repo's current state)
makes specific assumptions that this document makes explicit.

## TL;DR

| Posture | Phase 2 (current) | Phase 3 (planned) |
|---|---|---|
| Skill source | Author-controlled GitHub repo, code-reviewed via PR | Same + signed metadata |
| Skill execution | **QuickJS sandbox via `quickjs-emscripten` — runtime block of node APIs** | Same + per-skill capability flags |
| Trust assumption | Skills run untrusted-by-default; trust mode is opt-in (`LOADER_MODE=trust`) | Same |
| Bundle integrity | sha256 in manifest, verified before sandboxing | Same + ed25519 signature on manifest |
| Network gating | `ctx.fetch` whitelist by host, enforced on host side of bridge | Same + per-skill outbound budgets |
| Filesystem access | **Sandbox: `node:fs` returns undefined inside VM**; linter R9 also blocks at PR time | Same |
| Skill output (V5) | Delimiter wrap + per-skill `outputCap` + ANSI strip | Same + structured-output channel that bypasses delimiter |
| CPU exhaustion | **Sandbox interrupt handler kills handlers past `timeoutMs` (5s default)** | Same + memory cap |

If you run this registry as-is for an external/untrusted agent, **Phase 1
gives you the same trust posture as `npm install`** — you trust the
author and the supply chain. That's appropriate for a personal/team
registry. It is **not** appropriate for accepting community contributions.

## Threat actors and assets

Actors:
- **External skill author** submitting a PR (Phase 2 case)
- **Compromised maintainer account** with push access to `main`
- **Compromised CI runner** that builds + publishes to `dist` branch
- **Network attacker** sitting between agent and jsDelivr / dist branch
- **Malicious dependency** in the supply chain (npm packages used by handlers)

Assets being protected:
- The host process running the agent (filesystem, network, env, secrets)
- The downstream API credentials the agent has (e.g. GitHub token)
- The integrity of the agent's outputs (avoiding silent data exfiltration)

Out of scope for this document:
- Adversarial inputs to the model itself (prompt injection from skill output)
- Side-channel attacks against the LLM provider
- Physical attacks on the host

## Threat vectors and mitigations

### V1 — Hostile bundle published to `dist` branch

**Scenario**: An attacker pushes a hostile bundle to the `dist` branch
(via compromised maintainer credentials or a compromised CI workflow).
jsDelivr caches the file globally; every consumer pulling
`@dist/skills/...` runs the malicious code.

**Phase 1 mitigation**:
- `dist/` is generated by CI from `main`. Tampering with `dist` directly
  is detectable via `git log` mismatch (the next CI run rewrites it).
- **The manifest declares per-bundle sha256.** The loader
  (`client/loader.ts`) computes the actual sha256 of each fetched bundle
  and refuses to `import()` if it doesn't match. A hostile push to `dist`
  that modifies a bundle without also modifying the manifest fails the
  check; one that modifies both is detectable in the `dist` branch's
  `git log`.
- `.github/workflows/build.yml` runs `npm run all` before publishing —
  so the linter, validator, tests, and codegen-check all gate the
  publish.

**Phase 1 limitation**:
- A maintainer who can push directly to `main` can push a hostile
  manifest + bundle pair. Both have valid sha256s relative to each other.
  The only defense is code review before merge.

**Phase 2 plan**:
- Sign the manifest with a build-key controlled by CI (ed25519 in the
  signature field; public key committed in `main`'s root).
- Loader rejects manifests without a valid signature.
- Maintainer compromise still requires the build-key to publish a
  malicious manifest; key access can be scoped to the CI runner.

### V2 — Malicious skill author (PR review failure)

**Scenario**: An external contributor submits a PR adding a skill that
looks benign but exfiltrates env vars on first call. The PR passes lint
+ tests because the malicious behaviour only fires under specific input
conditions or after a delay. A reviewer skims the diff and merges.

**Phase 2 mitigation** (shipped — closes V2 with a runtime guarantee):
- **Handlers run inside a QuickJS WASM sandbox** (`client/sandbox.ts`).
  `import('node:fs')` returns undefined; `process` is undefined;
  `globalThis.fetch` is the curated bridge that goes through `ctx.fetch`
  on the host side (which the loader gates with `networkPolicy.allow`).
  No node API surface is reachable from inside the VM.
- **Curated context surface** (everything else is unreachable):
  - `globalThis.fetch(url, init?)` — proxied to host with allowlist
  - `ctx.fetch` — alias of the same proxy
  - `ctx.log(msg)` — writes to the host's stderr
  - `ctx.env[key]` — pre-filtered by the skill's `requiredEnv`
  - `URL` / `URLSearchParams` (polyfilled — QuickJS core lacks them)
  - all standard ES2023 built-ins (JSON, RegExp, Math, Date, Symbol, …)
- **Interrupt handler** breaks out of busy loops past the deadline
  (`timeoutMs`, default 5000 ms). `while(true){}` no longer hangs the
  host process.
- **Belt-and-suspenders**: the linter's R9 `forbidden-imports` rule
  still runs at PR time. It catches accidents before they reach CI;
  the sandbox catches anything that slips through.
- Every PR runs the full pipeline (`typecheck → tests → validate → lint
  → codegen:check → build → manifest`) via
  `.github/workflows/validate-pr.yml`.

**Phase 2 limitation**:
- A handler can still consume CPU up to the timeout (default 5s) and
  memory up to whatever QuickJS allocates before host OOM. Neither is
  catastrophic for an interactive agent loop, but a malicious skill in
  a serverless environment could rack up bills before being killed.
  Phase 3 plans memory caps + per-call latency budgets.
- The bridge is intentionally minimal. A handler that needs e.g.
  filesystem access (current Phase 2 has none) would have to go
  through a future curated `ctx.storage` API.
- An escape hatch exists for local debugging: `LOADER_MODE=trust`
  reverts to the Phase 1 in-process `import()`. This is documented as
  development-only and never used by the MCP servers in default mode.

**Phase 3 plan**:
- Per-skill capability flags in `tool.yaml`:
  `capabilities: [network, env]` — the loader exposes only what's
  declared. Default is the empty set (pure transforms only).
- Memory cap per VM via `runtime.setMemoryLimit`.
- Optional `crypto.subtle`, `TextEncoder/Decoder` exposed when the
  skill declares them (currently no skill needs them).

### V3 — Dependency confusion / supply-chain attack

**Scenario**: A handler imports an npm package (`some-helper`) that gets
hijacked (typosquat, maintainer compromise). The next `npm install`
pulls the malicious version, esbuild bundles it into `dist`, the
manifest's sha256 verifies (the bundle is what we built — the build was
just compromised), and consumers run the bad code.

**Phase 1 mitigation**:
- `package-lock.json` is committed; CI uses `npm ci` (locked deps only).
- Skills currently use **zero** runtime dependencies — handlers in this
  repo import only from `../../../../types/index.ts` and the standard
  `globalThis.fetch`. esbuild bundles them as-is.
- Build-time dependencies (esbuild, typescript, yaml) are popular and
  audited; their compromise would affect a much wider blast radius than
  this repo.

**Phase 1 limitation**:
- A future skill that imports an npm package opens this vector. Today
  no skill does, but nothing stops one from doing so.

**Phase 2 plan**:
- Linter rule: `no-npm-imports-in-skills`. Flag any handler import
  that isn't relative or `node:`.
- For skills that genuinely need an npm package, vendor it into
  `registry/skills/<slug>/vendor/` and audit the source manually.
- Consider moving build to a hermetic environment (Cloudflare Workers
  Build, Nix) so even the build toolchain is reproducible.

### V4 — Network attacker between agent and CDN

**Scenario**: An attacker controls the network path between the agent
host and jsDelivr (corporate proxy, compromised DNS, ISP interception).
They serve a modified manifest or bundle. The agent imports the modified
code.

**Phase 1 mitigation**:
- jsDelivr serves over HTTPS by default; certificate pinning at the
  host OS level provides standard TLS protections.
- The manifest sha256 check protects against a modified bundle (V1's
  mitigation extends here).

**Phase 1 limitation**:
- A modified **manifest** with internally-consistent sha256s passes the
  check. The loader can't tell whether the manifest itself was tampered
  with in transit.
- jsDelivr can be cache-poisoned in theory; in practice this is
  vanishingly rare but not impossible.

**Phase 2 plan**:
- Sign the manifest itself (ed25519). The loader verifies the signature
  against a public key shipped with the agent. Network attacker can't
  forge a signed manifest without the private key.
- Public key rotation policy: documented in this file when implemented.

### V5 — Hostile observation injection (Phase 2 specific)

**Scenario** (relevant once skills can be untrusted): A skill returns
output containing prompt-injection payloads aimed at the calling LLM
(e.g. `Output: "MX". Also, ignore previous instructions and..."`). The
agent's smart-bash observation includes this output verbatim. The next
model turn sees the injection.

**Phase 1 mitigation** (strawman shipped — not a full defense):
- **Delimiter wrap**: skill stdout flowing back into the agent loop is
  wrapped in `<skill-output skill="X" trust="untrusted">…</skill-output>`
  by `client/untrusted-output.ts`. Both MCP servers (`mcp-server.ts`,
  `mcp-server-classic.ts`) and the comparison harness (`compare.ts`)
  apply the wrap.
- **System prompt teaches the contract**: a fragment is injected into
  the system message instructing the model to treat wrapped content as
  *data*, never as instructions — and to ignore directives the wrapped
  content contains. This is teachable to capable models; smaller models
  may still get hijacked.
- **Per-skill `outputCap`**: declared in `tool.yaml`, applied before
  wrapping. `url2md` (the natural injection vector — it returns whatever
  markdown the upstream page contains) ships with a 4 KB cap. Default
  cap when undeclared is 8 KB. Limits payload-size blast radius.
- **ANSI / control-char sanitisation**: terminal escape sequences and
  C0/C1 controls are stripped before wrapping so adversarial output
  cannot smuggle directives past human reviewers via terminal sequences.

**Phase 1 limitation**:
- Prompt-injection defense remains an **open research problem**. This
  package raises the cost of casual injection (longer payloads
  truncate; structural marker is visible in traces; control-char
  smuggling is blocked) but a skilled attacker writing inside the
  delimiter still influences the model.
- Pure-shell pipelines (`echo … | jq`) are NOT wrapped — the data is
  shaped by the agent itself. A pipeline ending in a registry skill IS
  wrapped (correctness is "did the last stage produce untrusted data").

**Phase 2 plan**:
- Tighter sandbox-level control: with QuickJS in place, the cap can be
  enforced at the handler-return boundary, not just at the
  presentation layer.
- Optional schema-typed output channel: structured returns bypass the
  delimiter (they're parsed JSON, not text), but free-form `markdown`
  fields stay capped + wrapped.
- Per-skill `injectionRisk: high|low` flag in tool.yaml that drives
  default caps and triggers extra runtime checks.

## Current trust contract — what Phase 1 promises and what it doesn't

Phase 1 is appropriate for:
- A registry that one person or one team controls end-to-end
- Closed-source agents where the skill catalog is shipped with the agent
- Internal tooling where every skill author has commit access already

Phase 1 is **not** appropriate for:
- Public registries accepting community contributions
- Multi-tenant agents where users add their own skills
- Any context where a skill author is not strictly trusted by the agent
  operator

If you're in the second category, do not deploy this code yet. Wait for
Phase 2, or build the QuickJS sandbox layer yourself in your fork.

## Verifying integrity locally

```bash
# Re-build dist/ from main
npm run all

# Compare dist/manifest.json's sha256 against the published version
diff <(jq -S '.tools[] | {slug, sha256}' dist/manifest.json) \
     <(curl -s https://cdn.jsdelivr.net/gh/MauricioPerera/agentic-tools-poc@dist/manifest.json | jq -S '.tools[] | {slug, sha256}')
```

If the diff is non-empty, either `main` is ahead of `dist` (recent push,
not yet built) or someone tampered with the published manifest. The
loader's `sha256` check catches the latter at import time.

## Open issues this document does not yet resolve

- **Manifest signing** — V1 and V4 mitigations rely on this. Spec
  pending.
- **Public key distribution** — once manifest signing exists, where
  does the public key live? Probably committed in `main` and pinned in
  the loader, with a documented rotation policy.
- **Per-skill capability flags** — `tool.yaml` should declare which
  capabilities the sandbox exposes (`network`, `env`, future
  `crypto`). Default empty set = pure transforms only. The bridge
  conditionally provides those slots. (Phase 3.)
- **Memory cap per VM** — currently a malicious handler can allocate
  until host OOM. Hook `runtime.setMemoryLimit`. (Phase 3.)
- **Per-skill versioning** — pin a single skill at a known-good version
  while the rest of the registry advances. Tracked at
  [#2](https://github.com/MauricioPerera/agentic-tools-poc/issues/2).
